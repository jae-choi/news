대규모 데이터 분석 NL2SQL

NL2SQL for Large-Scale Data Analysis

Customer Problem Solving Case 2

“동적 패턴 캐싱과 자가 수정이 결합된 강건한 쿼리 엔진”

“Robust Query Engine with Dynamic Pattern Caching and Self-Correction”

비개발자 현업 부서가 거대한 BigQuery 데이터를 자연어로 직접 조회하려 할 때 발생하는 LLM 추론 지연과 문법 오류(Syntax Error)를 어떻게 시스템 구조로 흡수하여 안정적으로 런타임에 서비스할 수 있는지 다룬 설계 사례입니다.

This design case covers how to absorb LLM inference latency and syntax errors into the system structure to provide stable runtime services when non-developer departments try to directly query massive BigQuery data using natural language.

1. Need: 도입 배경

1. Need: Background

기획, 마케팅 부서가 수억 건의 결제/로그 데이터를 바탕으로 의사결정을 내리기 위해서는 데이터 분석가의 SQL 작성을 기다려야만 하는 병목이 있었습니다.

현업에서 일상적인 자연어 질문만 던지면 즉시 SQL로 변환하여 데이터를 시각화해주는 자연어-데이터 인터페이스(NL2SQL) 도입이 시급했습니다.

There was a bottleneck where planning and marketing departments had to wait for data analysts to write SQL queries to make decisions based on hundreds of millions of payment/log data records.

It was urgent to introduce a Natural Language-to-Data interface (NL2SQL) that would immediately convert everyday natural language questions from the field into SQL and visualize the data.

2. Problem Definition: 문제 정의

2. Problem Definition: Problem Statement

LLM에게 단순히 "이 질문을 SQL로 바꿔줘"라고 요청하면, 기업 고유의 복잡한 스키마를 잘못 해석하여 실행 불가능한 쿼리(Syntax Error)를 반환하는 빈도가 매우 높았습니다.

또한 매번 무거운 LLM 추론 API를 호출하느라 응답이 지연되고 비용이 낭비되는 문제가 있었습니다.

When simply asking the LLM to "convert this question to SQL," the frequency of it misinterpreting the enterprise's complex, proprietary schemas and returning unexecutable queries (Syntax Errors) was very high.

Additionally, calling heavy LLM inference APIs every time caused response delays and wasted costs.

3. Key View: 접근 방식

3. Key View: Approach

LLM의 1-Pass 생성을 100% 신뢰하지 않고, 생성된 SQL이 데이터베이스에서 에러를 뱉으면 그 에러 메시지를 LLM에게 다시 먹여 스스로 수정하게 하는 자가 수정(Self-Correction) 루프를 에이전트에 탑재했습니다.

더불어, LLM API 의존도를 낮추기 위해 이전에 성공한 SQL 변환 패턴을 인메모리에 저장해두고, 새로운 질의가 기존 의도와 유사하면 LLM 없이 변수만 치환하여 반환하는 동적 패턴 매칭(Dynamic Pattern Caching) 계층을 앞단에 배치했습니다.

Instead of trusting the LLM's 1-Pass generation 100%, we equipped the agent with a Self-Correction loop that feeds any database execution errors back into the LLM to fix itself.

Furthermore, to reduce reliance on the LLM API, we stored previously successful SQL conversion patterns in-memory. If a new query is similar to an existing intent, we placed a Dynamic Pattern Caching layer at the front that swaps only variables and returns the SQL without hitting the LLM.

4. Design / Framing: 아키텍처 설계

4. Design / Framing: Architecture Design

Cloud Run에 배포된 FastAPI 봇이 사용자 요청을 수신합니다. 먼저 Redis(K-Means 유사도 군집 캐시)를 조회하여 패턴이 맞으면 즉시 반환하고, 캐시 미스 시에만 Gemini LLM에 요청합니다.

Gemini가 만든 쿼리가 BigQuery에서 실행 실패하면, Validator 노드가 실패 원인을 Gemini로 다시 보내 쿼리를 재작성(Correction)시킵니다. 이 다단계 오케스트레이션을 통해 런타임의 치명적 중단을 막았습니다.

A FastAPI bot deployed on Cloud Run receives user requests. It first checks Redis (K-Means similarity cluster cache). If the pattern matches, it returns immediately; it requests the Gemini LLM only on a cache miss.

If the query generated by Gemini fails to execute in BigQuery, the Validator node sends the failure cause back to Gemini to rewrite the query (Correction). Through this multi-step orchestration, we prevented critical runtime interruptions.

5. Impact: 적용 결과

5. Impact: Results

이 아키텍처는 데이터 분석의 병목을 없애고 조직의 진정한 데이터 민주화를 가속했습니다.

This architecture eliminated data analysis bottlenecks and accelerated true data democratization in the organization.

  • ✓
    Runtime Stability Guaranteed Thanks to the Self-Correction loop, even if syntax errors occur in the initial 1-Pass generation, correct final data is delivered to the user.
  • ✓
