엔터프라이즈 RAG 챗봇 아키텍처

Enterprise RAG Chatbot Architecture

Customer Problem Solving Case 1

“단순 RAG를 넘어, 의도를 식별하는 하이브리드 라우팅 아키텍처”

“Beyond Simple RAG: Hybrid Routing Architecture for Intent Identification”

엔터프라이즈 사내 검색 환경에서는 단순한 키워드 매칭을 넘어, 다양한 문맥이 섞인 복합 질의를 유연하게 처리해야 합니다. 모든 요청을 단일 거대 모델에 위임하여 지연과 비용을 유발하는 대신, 앞단에 가벼운 의도 분류 모델(Intent Classifier)을 두어 검색 요청을 분기 처리하는 최적화 사례입니다.

In an enterprise internal search environment, it is necessary to go beyond simple keyword matching and flexibly handle complex queries mixed with various contexts. Instead of delegating all requests to a single giant model, which causes delays and costs, this is an optimization case that branches search requests by placing a lightweight Intent Classifier at the front.

1. Need: 도입 배경

1. Need: Background

기업 내 검색 요구사항이 점차 고도화됨에 따라 단순 정보 탐색을 넘어 여러 맥락이 얽힌 질문들이 쏟아지기 시작했습니다. 하지만 초기 구축된 단일 RAG(검색 증강 생성) 시스템으로는 이러한 복합적인 질의(문맥 혼재)를 효과적으로 처리하는 데 한계가 명확했습니다.

As internal search requirements became more sophisticated, complex questions intertwining multiple contexts began to pour in, going beyond simple information retrieval. However, the initial single RAG (Retrieval-Augmented Generation) system clearly had limits in effectively processing these complex, mixed-context queries.

2. Problem Definition: 문제 정의

2. Problem Definition: Problem Statement

가장 큰 문제는 질의의 '의도(Intent)'를 시스템이 명확하게 이해하지 못한 채 무조건적으로 전체 문서 DB를 뒤진다는 것이었습니다.

이로 인해 엉뚱한 문서가 검색 결과로 편입되거나 불필요한 LLM 추론 자원이 소모되는 비효율성이 발생했고, 결과적으로 오답률이 오르고 응답 지연(Latency)이 늘어나는 병목 현상에 직면했습니다.

The biggest problem was that the system unconditionally searched the entire document database without clearly understanding the 'intent' of the query.

This led to the inefficiency of irrelevant documents being included in the search results or unnecessary LLM inference resources being consumed. As a result, we faced bottlenecks with increasing incorrect answer rates and response latency.

3. Key View: 접근 방식

3. Key View: Approach

단순 RAG 시스템의 한계를 극복하기 위해, 텍스트를 생성하거나 문서를 검색하기 전 사용자의 질의 특성을 먼저 분류하여 가장 적합한 전담 에이전트(Agent)로 라우팅하는 구조가 필요하다고 판단했습니다.

검색의 품질을 좌우하는 것은 모델의 파라미터 크기가 아니라, 질의의 맥락에 맞춰 사내 문서(BigQuery)를 Lexical + Semantic 하이브리드 방식으로 정확히 연결해주는 '분류와 조율' 계층에 있다고 보았습니다.

To overcome the limitations of the simple RAG system, we judged that a structure was needed to first classify the characteristics of the user's query and route it to the most appropriate dedicated agent before generating text or searching documents.

We saw that search quality was determined not by the parameter size of the model, but by the 'classification and orchestration' layer that accurately links internal documents (BigQuery) in a hybrid Lexical + Semantic manner according to the context of the query.

4. Design / Framing: 아키텍처 설계

4. Design / Framing: Architecture Design

거대 단일 모델(Heavy LLM)에 모든 연산을 의존하여 생기는 고비용·고지연 문제를 회피하기 위해, 가벼운 의도 분류 모델(Lightweight Classifier 계층)을 앞단으로 분리(Decoupling)했습니다.

이 Classifier는 질의를 분석한 뒤, 적절한 RAG Agent(Vertex AI Agent Builder 기반 하이브리드 검색)로 분기시킵니다. 질의 로그와 사용자 세션은 SQL에서 관리하고 핵심 검색 대상 문서는 BigQuery에 적재하여, 성능과 정확도의 균형을 최적화했습니다.

To avoid the high-cost and high-latency issues caused by relying on a heavy single model for all computations, we decoupled a lightweight intent classification model to the front end.

This classifier analyzes the query and branches it to the appropriate RAG Agent (hybrid search based on Vertex AI Agent Builder). By managing query logs and user sessions in SQL and loading core search target documents into BigQuery, we optimized the balance between performance and accuracy.

5. Impact: 적용 결과

5. Impact: Results

분류기를 통해 사전 정제된 질의만 LLM과 검색 엔진에 도달하게 함으로써, 불필요한 API 비용을 삭감하고 응답 품질을 안정화했습니다.

By ensuring that only pre-refined queries reach the LLM and search engine via the classifier, we cut unnecessary API costs and stabilized response quality.
