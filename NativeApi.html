Native Multimodal 기반 음성 챗봇 아키텍처

STT 변환 과정 없이 오디오를 직접 이해하는 Gemini 2.5 Flash Native API를 도입하여, 5만 명 규모의 B2C 서비스에서 레이턴시를 없애고 인프라 비용을 85% 이상 절감한 아키텍처 시뮬레이션입니다.

5만 명의 사용자가 일일 평균 15분, 월 3시간의 한도(Quota) 내에서 실시간으로 대화형 AI를 이용하는 B2C 음성 챗봇 서비스를 설계해야 했습니다.

이 문서는 기존의 무거운 STT(Speech-to-Text) 파이프라인이 유발하는 막대한 API 과금과 응답 지연(Latency) 문제를, 오디오 네이티브(Native Audio) 모델 채택을 통해 어떻게 아키텍처 관점에서 최적화했는지 분석한 시뮬레이션 기록입니다.

1. Problem

레거시 STT 파이프라인의 예산 초과와 감정 손실

통제 불가능한 STT 비용 병목

사용자의 음성을 텍스트로 변환하여 LLM에 전달하는 전통적인 3단계(STT → LLM → TTS) 방식을 채택할 경우, 5만 명이 한도를 채워 사용할 때 STT(음성 인식) 비용만 연간 14만 4천 달러(약 2억 원)가 산출됩니다. 이는 전체 시스템 예산 상한선(Total Cap)을 심각하게 초과하는 수치입니다.

컨텍스트(감정, 억양) 누락 및 지연

STT API를 통과하면 사용자의 미세한 목소리 떨림이나 감정 상태 같은 오디오 고유의 컨텍스트(Context)가 평면적인 텍스트로 증발합니다. 또한 네트워크를 두 번(STT → LLM) 타면서 생기는 수 초의 지연 시간은 실시간 대화형 서비스의 몰입감을 크게 해칩니다.

2. Solution

Native Multimodal Audio 파이프라인 전면 도입

STT 미들웨어를 완전히 걷어내고, 오디오 스트림(Audio Stream) 자체를 직접 입력받아 이해하는 Gemini 2.5 Flash Native API를 아키텍처의 중심에 배치했습니다.

멀티모달 네이티브 모델이 소리를 텍스트 우회 없이 직접 분석함으로써, 사용자의 감정 톤을 잃지 않으면서 캐릭터 페르소나에 100% 동기화된 대답을 초저지연(Ultra-low Latency)으로 생성할 수 있게 되었습니다.

4. Technical Decisions

대규모 트래픽(Scalability) 방어 및 비용 최적화를 위한 아키텍처 전략

순수 프롬프팅 vs Context Caching 활용

캐릭터 챗봇은 말투, 세계관, 금지어 등 방대한 페르소나 설정값(System Prompt)을 매 턴마다 모델에 전달해야 합니다. 이는 막대한 입력 토큰 비용 누수로 이어집니다.

해결: 프롬프트 명시적 캐싱 도입 캐릭터 설정이 2,048 토큰을 넘어가는 경우, 이를 구글 클라우드 인프라 레벨에서 명시적으로 캐싱(Context Caching)하여 입력 비용을 90% 이상 절감하고 모델의 첫 번째 토큰 응답 시간(TTFT)을 극도로 단축시켰습니다.

Gemini 3.0 Flash vs 2.5 Flash 모델 선택

3.0 버전이 더 진보된 모델이긴 하나, 프리뷰 단가 정책상 2.5 Flash 대비 텍스트/오디오 입출력 단가가 10~20% 더 높게 책정되어 B2C 트래픽 규모에서는 상당한 예산 차이를 유발합니다.

해결: 가성비와 페르소나가 검증된 2.5 Flash 메인 배포 5만 명 동시 접속(Total Cap 시나리오)에서 무음 감지 로직(VAD)과 함께, 현존 최고 수준의 가성비를 가진 2.5 Flash를 프로덕션 메인 모델로 확정하여 예산 최적화와 품질 사이의 밸런스를 잡았습니다. 또한 Global Endpoints 및 PT(예약 용량)를 사전에 할당받아 트래픽 폭주 시의 429 에러(Too Many Requests)를 원천 차단했습니다.

5. Business Impact

인프라 운영 비용 85% 파격 절감

시뮬레이션 결과, 사용자 5만 명이 부여된 한도(3시간)를 100% 소모했을 때 기존 STT 방식의 연간 예산 상한선 2.01억 원을 2,520만 원 수준으로 약 85% 압축했습니다.

비용을 줄이면서도 UX는 혁신하다

  • 직관적 감정 교류 (Zero-Latency): 두 번의 무거운 네트워크 점프(STT, LLM)를 하나로 합쳐 지연 시간을 소거했습니다. 사람과 대화하듯 반응하는 캐릭터를 통해 몰입도 높은 프로덕트 완성이 가능합니다.
  • 스케일 아웃 설계 완료: 단일 LLM에 종속되지 않고 Cloud Run 기반의 Global Endpoints로 앱 레이어를 구축해두어, 추후 수십만 단위 트래픽으로 서비스가 확장되어도 즉각적인 대응이 가능합니다.
