단일 스크립트 기반의 AI 모델 테스트는 구축이 간편하지만, 다수의 모델과 에이전트가 복잡하게 상호작용하는 환경에서는 제어가 어렵습니다.

특히 내부 의사결정 과정(Thought Process) 추적, 리소스 비용 모니터링, 그리고 시스템 확장 등 운영 관점에서의 요구사항을 만족시키기 위해서는 별도의 시스템 아키텍처가 필요합니다.

`observe-llm`은 이러한 배경에서 시작된 프로젝트로, 실무 수준(Production-Ready)의 AI 모니터링 및 로깅 파이프라인을 검증하기 위한 레퍼런스 구현체입니다.

1. Problem

블랙박스가 되어버리는 복잡한 AI 에이전트 워크플로우의 한계

단방향 생성의 구조적 결함

대부분의 단순 LLM 애플리케이션은 질문을 던지고 답변을 받는 '1-pass' 체인에 의존합니다.

문제: 스스로 오류를 발견하고 수정할 수 없어, 복잡한 태스크에서 환각(Hallucination)과 논리적 비약이 통제 없이 노출됨

자율 에이전트의 '블랙박스'화

이를 해결하기 위해 순환적인 루프(자기 비판/수정)를 도입하면, 반대로 에이전트 내부에서 어떤 결정이 어떻게 내려졌는지 개발자가 파악하기 불가능해집니다.

문제: 오류 발생 시 원인을 추적할 수 없고(디버깅 불가), 병목 구간(Latency) 파악 및 토큰 비용 측정이 어려움

2. Solution

LangGraph와 Langfuse를 연동한 추적 가능한 에이전트 파이프라인 설계

콘텐츠 검수 및 보완 프로세스를 구조화하여, '작성 → 비평 → 수정'을 시스템이 자체적으로 반복하는 상태 기반(State-based) 워크플로우를 LangGraph로 구축했습니다.

이 과정에 Langfuse 플랫폼을 연동하여, 에이전트의 세부 실행 내역(Trace)과 노드별 의사결정 과정을 시각화(Observability)하도록 설정했습니다.

Self-Correcting 워크플로우 구성

1 초안 작성 (Draft)

지정된 주제 혹은 제공된 데이터를 기반으로 초기 콘텐츠 초안을 생성합니다.

2 논리 검증 (Critique)

사전 정의된 평가 기준에 따라 작성된 초안의 논리적 모순이나 정보 누락 여부를 검토합니다.

3 동적 분기 및 기록 (Routing & Logging)

검증 결과에 따라 수정이 필요하면 'Reviser'로 이동하고, 기준을 충족하면 프로세스를 종료합니다. 이 라우팅 과정 전체의 메타데이터와 소요 시간은 Langfuse에 자동 기록됩니다.

3. Architecture

에이전트를 지원하는 전체 운영 생태계(Ecosystem)의 구축

견고한 시스템은 단순한 파이썬 스크립트 하나로 완성되지 않습니다. API 요청부터 모니터링 로깅, 데이터 저장까지 전체 스택이 원활하게 조율되어야 합니다.

4. Technical Decisions

안정적 운영과 인프라 확장을 위해 거쳤던 치열한 엔지니어링 고민

단순 파이프라인 vs 조건부 엣지(Conditional Edge) 라우팅

결과가 나올 때까지 기다리기만 하는 선형적 파이프라인은 구축이 쉽지만, 답변의 논리적 오류를 방어할 메커니즘이 없습니다.

해결: 조건부 엣지(Conditional Edge) 설계 AgentState 객체를 활용하여 노드 간 데이터를 공유하고, 라우팅 함수에서 출력물 내 특정 문자열(예: 'REVISE') 포함 여부를 검사하여 조건부 흐름 제어를 구현했습니다. 
def should_revise(state: AgentState) -> str:
    if "REVISE" in state["critique"]:
        return "reviser"  # Loop back to revise
    else:
        return "set_final_output" # Exit the loop

빠른 데모 세팅 vs 영속성(Persistence)과 보안 중심 인프라

테스트 목적의 컨테이너 환경에서는 종료 시 데이터가 소멸되며, 구성 정보(API 키 등)가 코드 내에 남아있을 경우 운영 환경 배포 시 취약점이 됩니다.

해결: 볼륨 바인딩 및 구성 파일 격리 로그 DB(PostgreSQL, Clickhouse) 컨테이너에 로컬 바인드 마운트를 적용하여 시스템 재시작 시에도 로깅 데이터를 보존합니다. 더불어 .env 파일 기반의 환경 변수 주입 구조를 통해 민감 정보를 소스코드로부터 물리적으로 분리했습니다.

5. Business Impact

가시성 기반의 서비스 운영 통제력 확보

모델별 응답 속도 지연(Latency) 원인과 노드 간 에러 구간을 시각적으로 모니터링할 수 있습니다. 또한, 사용자 ID 단위(user_id)의 토큰 소비량 로깅 체계를 구축하여, API 비용 분석 및 B2B 서비스 과금 시스템 연동을 위한 데이터 기반을 마련했습니다.

정량적 모델 평가 체계 구축

  • 프롬프트 A/B 테스트 환경: 동일한 입력 데이터에 대해 버전별 프롬프트의 결과 품질을 비교 및 수치화하여, 애플리케이션 업데이트 시 발생할 수 있는 품질 저하 리스크를 최소화합니다.
  • 추가 도구 연동 유연성: 해당 모니터링 아키텍처는 향후 웹 검색 플러그인(Tavily 등)이나 외부 RAG 데이터베이스를 확장 연동하더라도, 동일한 뷰 내에서 실행 로그를 통합 추적할 수 있도록 설계되었습니다.
