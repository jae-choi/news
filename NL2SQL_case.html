대규모 데이터 분석 NL2SQL 아키텍처

NL2SQL Architecture for Large-Scale Data Analysis

Customer Problem Solving Case 2

“자연어 질문을, 운영 가능한 SQL 추론 엔진으로 연결하다”

“Connecting natural-language questions to an operable SQL inference engine”

비개발 부서가 대규모 데이터를 활용하려면 늘 SQL 작성 지원이 필요했습니다. 이 사례는 자연어 질문을 SQL과 시각화 결과로 연결하되, 단순 생성이 아니라 자가 수정(Self-Correction)과 동적 패턴 캐싱(Dynamic Pattern Caching)을 결합한 운영형 NL2SQL 구조로 분석 병목을 줄인 설계 브리프입니다.

Non-technical teams often needed analyst support whenever they wanted to use large-scale data. This case brief shows how natural-language questions were connected to SQL and visualization outputs through an operational NL2SQL architecture combining self-correction and dynamic pattern caching, rather than relying on one-shot generation alone.

1. Need: 도입 배경

1. Need: Background

기업 내 데이터 활용 요구가 고도화되면서, 단순 대시보드 조회를 넘어 현업 부서가 직접 질문을 던지고 즉시 답을 얻을 수 있는 구조가 필요해졌습니다.

하지만 대규모 데이터 환경에서는 여전히 SQL 작성 역량이 진입 장벽으로 작동했고, 기획, 마케팅, 운영 부서의 질문은 데이터 분석가를 거쳐야만 처리되는 병목으로 남아 있었습니다.

As enterprise demand for data access grew more sophisticated, teams needed a structure that allowed business users to ask questions directly and get immediate answers beyond static dashboard lookup.

However, in large-scale data environments, SQL expertise still acted as a barrier, and planning, marketing, and operations teams had to rely on analysts, creating a persistent bottleneck.

2. Problem Definition: 문제 정의

2. Problem Definition: Problem Statement

단순 NL2SQL 파이프라인만으로는 사용자의 자연어 질의를 안정적으로 실행 가능한 SQL로 변환하는 데 한계가 있었습니다.

기업 고유의 스키마를 잘못 해석해 실행되지 않는 쿼리(Syntax Error)를 만들거나, 반복되는 유사 질의에도 매번 LLM을 다시 호출해 지연과 토큰 비용이 누적되는 문제가 발생했습니다.

A naive NL2SQL pipeline was not sufficient to convert natural-language questions into consistently executable SQL.

It frequently misread enterprise-specific schemas, produced invalid queries, and repeatedly called the LLM even for recurring queries, increasing latency and token cost.

3. Key View: 접근 방식

3. Key View: Approach

핵심은 자연어를 한 번에 SQL로 번역하는 것이 아니라, 실행 가능한 쿼리로 수렴시키는 추론 체계를 운영 관점에서 설계하는 것이라고 판단했습니다.

이를 위해 생성 이후 검증과 재작성 단계를 두는 자가 수정(Self-Correction) 루프와, 자주 반복되는 질의를 빠르게 흡수하는 동적 패턴 캐싱(Dynamic Pattern Caching)을 결합하는 것이 효과적이라고 보았습니다.

The key was not translating natural language into SQL in one shot, but designing an operational inference system that converges on an executable query.

To achieve this, we combined a self-correction loop that validates and rewrites generated SQL with dynamic pattern caching that absorbs recurring query patterns efficiently.

4. Design / Framing: 아키텍처 설계

4. Design / Framing: Architecture Design

모든 질의를 무거운 거대 모델에 그대로 위임하는 대신, 자연어 질의를 SQL로 생성한 뒤 Validator 계층에서 구문 오류를 감지하고 LLM으로 다시 되돌리는 Self-Correction 구조를 두었습니다.

또한 문장을 통째로 저장하는 단순 캐시 대신, 기간, 필터, 조회 범위 같은 SQL 변수 패턴을 추출해 저장하는 동적 패턴 캐싱 계층을 별도로 두어 성능과 응답 속도를 최적화했습니다. 쿼리 실행 대상은 BigQuery에 두고, 결과는 시각화 처리 후 캐시에 재적재하는 방식으로 반복 질의 대응력을 높였습니다.

Instead of delegating every query directly to a heavy model, we generated SQL first and placed a validator layer that detects syntax issues and routes failures back into the LLM through a self-correction loop.

We also introduced a dynamic pattern caching layer that stores reusable SQL variable patterns such as time ranges, filters, and scopes instead of caching entire sentences. Queries were executed against BigQuery, then visualization-ready results were stored back into the cache to improve repeated-query responsiveness.

5. Impact: 적용 결과

5. Impact: Results

반복되는 유사 질의에 대해 불필요한 LLM 호출을 줄이고, 실행 오류를 시스템 내부에서 흡수하면서 전반적인 응답 품질과 운영 안정성을 높였습니다.

By reducing unnecessary LLM calls for recurring queries and absorbing execution failures inside the system, we improved both response quality and operational stability.

  • ✓
